UV-langChain

安装

powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
uv --version
#查看可以版本
uv python list
#安装版本
uv python install 3.11.6
uv python install pypy3.10
#设置默认版本
uv python install 3.13.12 --default
# 输出 Python 3.12.x 即成功
python --version

虚拟环境

# 方式1：创建默认名称（.venv）的虚拟环境（推荐，符合行业规范）
uv venv

# 方式2：指定 Python 版本创建虚拟环境（如 3.11）
uv venv --python 3.11 .venv
uv venv --python=3.11 .venv # 两种写法均可

# 方式3：创建自定义名称的虚拟环境（区分不同项目）
uv venv --python 3.12 .venv312 # 适配 Python 3.12 项目

# 方式4：指定本地已安装的 Python 可执行文件路径（Windows 专用）
uv venv --python "C:\Python311\python.exe" .venv

激活虚拟环境

# macOS/Linux 系统（终端执行）
source .venv/bin/activate

# Windows 系统（CMD/PowerShell 执行）
.venv\Scripts\activate
.venv37\Scripts\activate # 自定义名称环境激活

安装

# 安装最新版本的包（如 requests）
uv pip install requests

# 安装指定版本的包（避免版本兼容问题）
uv pip install requests==2.31.0

# 从 requirements.txt 文件批量安装依赖（迁移项目必备）
uv pip install -r requirements.txt

# 安装包到开发环境（仅开发时使用，如测试工具 pytest）
uv pip install --dev pytest

# 升级指定包到最新版本
uv pip upgrade requests

# 升级所有已安装的包（谨慎使用，避免版本冲突）
uv pip upgrade --all

# 卸载指定包（彻底删除，无残留）
uv pip uninstall requests

推出虚拟环境

deactivate

清理缓存

uv cache clean

项目初始化（快速创建规范项目）

uv init my_project # 创建名为 my_project 的项目
cd my_project # 进入项目目录

简单来说：

  • ‌如果你在一个由 uv init 创建的标准项目中‌：uv run优先使用 ‌**uv add**‌ 声明的依赖（通过 uv.lock 锁定）。
  • ‌如果你只是手动创建了一个虚拟环境，没有项目配置文件‌：uv run 会使用当前虚拟环境中已安装的包，无论它是通过 uv pip install 还是其他方式安装的。

以下是详细的行为逻辑分析：

1. 场景一：标准项目管理模式（推荐）

uv python find 查询当前虚拟环境

‌特征‌：项目根目录有 pyproject.toml 和 uv.lock 文件。

在这种情况下，‌**uv run 严格遵循 uv.lock 文件中的依赖列表**‌。

  • ‌**uv add 的作用**‌：当你运行 uv add requests 时，uv 会解析依赖，更新 pyproject.toml（记录需求）和 uv.lock（记录精确版本和哈希值），并同步安装到 .venv。
  • ‌**uv pip install 的作用‌：如果你在这个模式下运行 uv pip install numpy，它确实会把 numpy 安装到 .venv 中，‌但是**‌：
    • 它‌不会‌更新 pyproject.toml。
    • 它‌不会‌更新 uv.lock。
  • ‌**uv run 的行为**‌：
    • 当你在该目录下运行 uv run python main.py 时，uv 会检查 uv.lock。
    • 如果下次你或你的同事运行 uv sync，‌通过 uv pip install 安装的 numpy 可能会被卸载‌，因为它不在 uv.lock 中。
    • 因此，在标准项目模式下，uv run 运行的环境状态是由 ‌**uv add + uv sync**‌ 决定的，而不是 uv pip。

‌结论‌：在标准项目中，请始终使用 uv add。uv pip install 安装的包是“临时”且“不受管理”的，随时可能在同步时被清除。

2. 场景二：非项目管理模式（传统虚拟环境）

‌特征‌：你手动运行了 uv venv 创建了一个虚拟环境，但‌没有‌运行 uv init，或者目录中‌没有‌ pyproject.toml 和 uv.lock。

在这种情况下，‌**uv run 仅仅是一个“在当前虚拟环境中运行命令”的工具**‌。

  • ‌行为‌：uv run 会激活当前的 .venv，然后执行后面的命令。
  • ‌依赖来源‌：它不关心依赖是怎么来的。只要包安装在 .venv/Lib/site-packages (Windows) 或 .venv/lib/pythonX.X/site-packages (Linux/Mac) 中，uv run 就能找到并使用它。
  • ‌此时 uv pip install 和 uv add 的区别‌：
    • 如果没有 pyproject.toml，uv add 通常会报错或提示你初始化项目。
    • 你主要使用 uv pip install 来安装包。
    • uv run 使用的就是这些通过 uv pip install 安装的包。

‌结论‌：在没有项目配置文件的简单脚本或测试环境中，uv run 使用的是当前虚拟环境中所有已安装的包（通常由 uv pip install 安装）。

3. 如何验证当前使用的是哪套机制？

你可以在终端运行以下命令来判断：

bash# 查看当前目录是否有锁文件
ls uv.lock pyproject.toml
  • ‌如果有这两个文件‌：你处于‌项目管理模式‌。
    • ✅ 正确做法：使用 uv add <package> 添加依赖。
    • ❌ 避免做法：不要混用 uv pip install，否则会导致环境不一致。
    • 🚀 uv run 将基于 uv.lock 确保环境的一致性。
  • ‌如果没有这两个文件‌：你处于‌临时/传统模式‌。
    • ✅ 做法：使用 uv pip install <package> 安装包。
    • 🚀 uv run 将直接调用当前虚拟环境中的 Python 解释器和已安装的包。

总结建议

为了保证项目的可复现性和稳定性，‌强烈建议采用场景一（项目管理模式）‌：

  1. 使用 uv init 初始化项目。
  2. 使用 ‌**uv add**‌ 添加所有依赖。
  3. 使用 ‌**uv run**‌ 运行代码。

这样，uv run 运行的依赖完全由 uv add 管理，并通过 uv.lock 锁定，确保在任何机器上运行结果一致。‌**尽量避免在已有 uv.lock 的项目中使用 uv pip install**‌，除非你非常清楚自己在做什么（例如临时调试某个特定版本的包，且不打算提交代码）。

在使用 uv 管理 Python 项目时，‌推荐先创建项目（初始化），再让 uv 自动处理虚拟环境‌。

实际上，在现代 uv 的工作流中，你通常不需要手动单独执行“创建虚拟环境”这一步，因为 uv init 命令会自动为你完成项目的初始化以及虚拟环境的创建。

以下是详细的操作逻辑和两种常见场景的对比：

1. 推荐工作流：先初始化项目（一站式完成）

这是 uv 官方最推荐的方式。当你运行 uv init 时，它会一次性完成以下所有操作：

  1. 创建项目目录结构。
  2. 生成 pyproject.toml（项目配置文件）。
  3. 生成 uv.lock（依赖锁定文件）。
  4. ‌自动创建并关联虚拟环境‌（通常在 .venv 目录下）。

‌操作步骤：‌

powershell# 1. 初始化项目（同时创建了虚拟环境）
uv init my_project

# 2. 进入项目目录
cd my_project

# 3. 添加依赖（uv 会自动安装到刚才创建的 .venv 中）
uv add requests

# 4. 运行代码（uv run 会自动使用 .venv 环境，无需手动激活）
uv run python main.py

‌优点：‌

  • ‌省心‌：无需手动运行 uv venv。
  • ‌规范‌：自动建立项目配置与环境的绑定关系。
  • ‌统一‌：依赖记录在 pyproject.toml 中，环境状态锁定在 uv.lock 中，便于团队协作和复现。

2. 传统/临时工作流：先创建虚拟环境（不推荐用于正式项目）

如果你只是想快速测试某个库，或者习惯传统的 venv + pip 模式，你可以选择先创建虚拟环境。但这通常用于‌非项目管理‌的场景。

‌操作步骤：‌

powershell# 1. 手动创建虚拟环境
uv venv

# 2. 激活虚拟环境（可选，uv pip  install 可以不激活直接装，但激活后更直观）
.\.venv\Scripts\Activate.ps1  # Windows PowerShell
# source .venv/bin/activate   # Linux/Mac

# 3. 使用兼容 pip 的命令安装包
uv pip install requests

# 4. 运行代码
python main.py

‌缺点：‌

  • ‌无项目配置‌：没有 pyproject.toml 和 uv.lock，依赖版本无法锁定，其他人难以复现你的环境。
  • ‌管理分散‌：需要手动管理环境激活和包安装，容易出错。

总结与建议

表格

场景 推荐顺序 核心命令 说明
‌正式项目开发‌ ‌先创建项目‌ uv init <name> uv init 会自动创建虚拟环境。后续使用 uv add 和 uv run 管理依赖和运行。
‌临时脚本/测试‌ 先创建环境 uv venv 仅用于快速实验，不使用项目配置文件。后续使用 uv pip install。

‌结论：‌
对于绝大多数情况，请直接使用 ‌**uv init <项目名>**‌。它不仅创建了项目，也 implicitly（隐式地）为你创建好了虚拟环境。你之后所有的依赖安装（uv add）和代码运行（uv run）都会自动在这个环境中进行，无需关心“先有鸡还是先有蛋”的问题。
