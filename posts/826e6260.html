概念

中断允许您在特定点暂停图执行，并在继续之前等待外部输入。这实现了需要外部输入才能继续的人机交互模式。当中断被触发时，LangGraph 使用其持久化层保存图状态，并无限期等待，直到您恢复执行。

中断通过在图节点的任何位置调用 interrupt() 函数来工作。该函数接受任何可 JSON 序列化的值，该值将呈现给调用者。当您准备好继续时，通过使用 Command 重新调用图来恢复执行，该命令随后成为节点内部 interrupt() 调用的返回值。

与静态断点（在特定节点之前或之后暂停）不同，中断是动态的：它们可以放置在代码中的任何位置，并且可以根据您的应用程序逻辑设置条件。

  • 检查点保留您的位置：检查点记录器会写入确切的图状态，以便您以后可以恢复，即使是在错误状态下也是如此。
  • thread_id 是您的指针：设置 config={"configurable": {"thread_id": ...}} 以告诉检查点记录器加载哪个状态。
  • 中断负载通过 chunk["interrupts"] 呈现：在使用 version="v2" 进行流式传输时，您传递给 interrupt() 的值会出现在 values 流部分的 interrupts 字段中，以便您了解图正在等待什么。

您选择的 thread_id 实际上是您的持久光标。重复使用它可以恢复同一个检查点；使用新值则会以空状态启动一个全新的线程。

使用 interrupt 暂停

interrupt 函数会暂停图执行并向调用者返回一个值。当您在节点中调用 interrupt 时，LangGraph 会保存当前的图状态并等待您通过输入恢复执行。要使用 interrupt，您需要：

  1. 一个检查点记录器 (checkpointer) 来持久化图状态（在生产环境中使用持久化检查点记录器）
  2. 配置中的 **线程 ID (thread ID)**，以便运行时知道从哪个状态恢复
  3. 在您想要暂停的地方调用 interrupt()（负载必须是可 JSON 序列化的）
from langgraph.types import interrupt

def approval_node(state: State):
    # Pause and ask for approval
    approved = interrupt("Do you approve this action?")

    # When you resume, Command(resume=...) returns that value here
    return {"approved": approved}
"""
LangGraph interrupt() 中断机制演示

interrupt() 用于在图执行过程中暂停，等待外部输入。
调用后图立即停止，返回当前状态，后续可通过 Command(resume=...) 恢复。
"""
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import interrupt, Command

# 场景1: 基础中断
# ============================================================

class State(TypedDict):
    action: str
    approved: bool
    result: str


def prepare_action(state: State):
    print("  [prepare_action] 准备操作中...")
    return {"action": "删除数据库表 users"}


def human_approval(state: State):
    print(f"  [human_approval] 需要审批: {state['action']}")
    decision = interrupt({
        "question": "是否允许执行以下操作？",
        "action": state["action"],
    })
    return {"approved": decision}


def execute_action(state: State):
    if state["approved"]:
        print(f"  [execute_action] ✅ 审批通过，执行: {state['action']}")
        return {"result": f"已执行: {state['action']}"}
    else:
        print(f"  [execute_action] ❌ 审批拒绝，跳过")
        return {"result": "操作已取消"}


g1 = StateGraph(State)
g1.add_node(prepare_action)
g1.add_node(human_approval)
g1.add_node(execute_action)
g1.add_edge(START, "prepare_action")
g1.add_edge("prepare_action", "human_approval")
g1.add_edge("human_approval", "execute_action")
g1.add_edge("execute_action", END)

checkpointer1 = InMemorySaver()
graph1 = g1.compile(checkpointer=checkpointer1)

print("=" * 60)

config1: RunnableConfig = {"configurable": {"thread_id": "demo-1"}}

当您调用 interrupt 时，会发生以下情况

  1. 图执行被挂起，就在调用 interrupt 的确切位置
  2. 状态被保存，使用检查点记录器以便以后可以恢复执行。在生产环境中，这应该是一个持久化检查点记录器（例如由数据库支持）
  3. 值被返回给调用者，位于 __interrupt__ 下；它可以是任何可 JSON 序列化的值（字符串、对象、数组等）
  4. 图无限期等待，直到您带着响应恢复执行
  5. 响应被传回到节点中，当您恢复时，它成为 interrupt() 调用的返回值

恢复中断

在中断暂停执行后，您通过再次调用图并包含 Command（其中包含恢复值）来恢复图。恢复值被传回给 interrupt 调用，允许节点使用外部输入继续执行。

from langgraph.types import Command

# Initial run - hits the interrupt and pauses
# thread_id is the persistent pointer (stores a stable ID in production)
config = {"configurable": {"thread_id": "thread-1"}}
result = graph.invoke({"input": "data"}, config=config, version="v2")

# result is a GraphOutput with .value and .interrupts
# .interrupts contains the payloads passed to interrupt()
print(result.interrupts)
# > (Interrupt(value='Do you approve this action?'),)

# Resume with the human's response
# The resume payload becomes the return value of interrupt() inside the node
graph.invoke(Command(resume=True), config=config, version="v2")

关于恢复的关键点

  • 恢复时必须使用与发生中断时相同的 线程 ID
  • 传递给 Command(resume=...) 的值成为 interrupt 调用的返回值
  • 恢复时，节点会从调用 interrupt 的节点开头重新开始，因此 interrupt 之前的任何代码都会再次运行
  • 您可以将任何可 JSON 序列化的值作为恢复值传递

常见模式

中断开启的关键功能是能够暂停执行并等待外部输入。这对于多种用例都很有用，包括：

  • 审批工作流：在执行关键操作（API 调用、数据库更改、财务交易）之前暂停
  • 处理多个中断：在单次调用中恢复多个中断时，将中断 ID 与恢复值配对
  • 审阅并编辑：在继续之前让人员审阅并修改 LLM 输出或工具调用
  • 中断工具调用：在执行工具调用之前暂停，以便审阅并编辑工具调用
  • 验证人工输入：在进行下一步之前暂停，以验证人工输入

结合人机交互 (HITL) 中断进行流式传输

在构建具有人机交互工作流的交互式智能体时，您可以同时流式传输消息块和节点更新，以便在处理中断时提供实时反馈。将多种流模式（"messages" 和 "updates"）与 subgraphs=True（如果存在子图）结合使用，可以：

  • 实时流式传输 AI 生成的响应
  • 检测图何时遇到中断
  • 无缝处理用户输入并恢复执行

"""
LangGraph: 人机交互 (HITL) 中断 + 流式传输

演示如何在流式输出中处理 interrupt 中断，
实现"边执行边输出，遇到审批暂停，人工确认后继续"的效果。
"""
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import interrupt, Command
from langgraph.types import StreamMode
from langchain_core.runnables import RunnableConfig
from langchain_core.messages import HumanMessage, AIMessage, AnyMessage
from operator import add


# ============================================================
# 定义状态
# ============================================================
class State(TypedDict):
    messages: Annotated[list[AnyMessage], add]
    pending_action: str
    action_result: str


# ============================================================
# 节点定义
# ============================================================
def chatbot(state: State):
    """AI 聊天节点：根据用户消息生成回复"""
    last_msg = state["messages"][-1].content if state["messages"] else ""

    if "删除" in last_msg or "drop" in last_msg.lower():
        reply = AIMessage(content="检测到危险操作，我需要先让你确认一下。")
        return {
            "messages": [reply],
            "pending_action": f"执行: {last_msg}",
        }
    else:
        reply = AIMessage(content=f"好的，我来处理: {last_msg}")
        return {"messages": [reply], "pending_action": ""}


def human_review(state: State):
    """人工审核节点：中断等待确认"""
    if not state["pending_action"]:
        return {"action_result": "无需审核"}

    decision = interrupt({
        "type": "approval",
        "message": "请确认是否执行以下操作？",
        "action": state["pending_action"],
    })
    return {"action_result": "approved" if decision else "rejected"}


def execute(state: State):
    """执行节点：根据审核结果执行"""
    result = state["action_result"]
    if result == "approved":
        msg = AIMessage(content=f"✅ 已执行: {state['pending_action']}")
    elif result == "rejected":
        msg = AIMessage(content=f"❌ 已取消: {state['pending_action']}")
    else:
        msg = AIMessage(content="操作完成")
    return {"messages": [msg], "pending_action": ""}


# ============================================================
# 路由逻辑
# ============================================================
def needs_review(state: State) -> str:
    """判断是否需要人工审核"""
    if state["pending_action"]:
        return "human_review"
    return END


def after_review(state: State) -> str:
    """审核后进入执行"""
    return "execute"


# ============================================================
# 构建图
# ============================================================
workflow = StateGraph(State)
workflow.add_node(chatbot)
workflow.add_node(human_review)
workflow.add_node(execute)

workflow.add_edge(START, "chatbot")
workflow.add_conditional_edges("chatbot", needs_review, {
    "human_review": "human_review",
    END: END,
})
workflow.add_edge("human_review", "execute")
workflow.add_edge("execute", END)

checkpointer = InMemorySaver()
graph = workflow.compile(checkpointer=checkpointer)

# ============================================================

工具中的中断

您也可以直接在工具函数内部放置中断。这使得工具本身在被调用时会暂停以待批准，并允许在执行之前对工具调用进行人工审阅和编辑。首先，定义一个使用 interrupt 的工具：

"""
LangGraph 工具中添加中断

场景：在工具执行前插入中断，等待人工确认后再执行工具。
常见用途：危险工具审批、工具参数确认、敏感操作授权。
"""
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import interrupt, Command
from langgraph.prebuilt import ToolNode
from langchain_core.runnables import RunnableConfig
from langchain_core.messages import HumanMessage, AIMessage, AnyMessage, ToolMessage
from operator import add
import random


# ============================================================
# 定义工具（工具内部包含中断）
# ============================================================

def search_database(query: str) -> str:
    """搜索数据库（安全操作，无需审批）"""
    results = f"搜索 '{query}' 找到 3 条结果: [文档A, 文档B, 文档C]"
    print(f"  [search_database] {results}")
    return results


def delete_records(table: str, condition: str) -> str:
    """删除记录（危险操作，需要人工确认）"""
    print(f"  [delete_records] ⚠️ 危险操作: 删除 {table} 中 {condition}")

    # 在工具内部中断，等待人工确认
    approved = interrupt({
        "type": "tool_approval",
        "tool": "delete_records",
        "args": {"table": table, "condition": condition},
        "warning": "此操作不可逆，是否确认执行？",
    })

    if approved:
        msg = f"已删除 {table} 中满足 {condition} 的记录"
        print(f"  [delete_records] ✅ {msg}")
        return msg
    else:
        msg = f"操作已取消: 删除 {table} 中 {condition}"
        print(f"  [delete_records] ❌ {msg}")
        return msg


def send_email(to: str, subject: str, body: str) -> str:
    """发送邮件（敏感操作，需要确认）"""
    print(f"  [send_email] ⚠️ 敏感操作: 发邮件给 {to}")

    approved = interrupt({
        "type": "tool_approval",
        "tool": "send_email",
        "args": {"to": to, "subject": subject, "body": body},
        "warning": "确认发送此邮件？",
    })

    if approved:
        msg = f"邮件已发送给 {to}: {subject}"
        print(f"  [send_email] ✅ {msg}")
        return msg
    else:
        msg = f"邮件发送已取消"
        print(f"  [send_email] ❌ {msg}")
        return msg


# 工具列表
tools = [search_database, delete_records, send_email]
tools_by_name = {t.__name__: t for t in tools}

class AgentState(TypedDict):
    messages: Annotated[list[AnyMessage], add]


def tool_node(state: AgentState):
    """工具节点：执行工具调用"""
    last_message = state["messages"][-1]
    results = []

    for tool_call in last_message.tool_calls:
        tool = tools_by_name[tool_call["name"]]
        print(f"\n  [tool_node] 调用工具: {tool_call['name']}({tool_call['args']})")

中断规则

当您在节点中调用 interrupt 时，LangGraph 通过抛出一个向运行时发出暂停信号的异常来挂起执行。此异常通过调用栈向上传播并由运行时捕获，运行时通知图保存当前状态并等待外部输入。

当执行恢复时（在您提供了请求的输入之后），运行时会从头开始重新启动整个节点——它不会从调用 interrupt 的确切行恢复。这意味着在 interrupt 之前运行的任何代码都会再次执行。因此，在使用中断时需要遵循一些重要规则，以确保它们的行为符合预期。

不要将 interrupt 调用包裹在 try/except 中

interrupt 在调用点暂停执行的方式是抛出一个特殊的异常。如果您将 interrupt 调用包裹在 try/except 块中，您将捕获此异常，中断将不会传回给图。

  • ✅ 将 interrupt 调用与易错代码分开
  • ✅ 在 try/except 块中使用特定的异常类型

不要在节点内重新排序 interrupt 调用

在单个节点中使用多个中断是很常见的，但如果不小心处理，可能会导致意外行为。当一个节点包含多个中断调用时，LangGraph 会维护一个特定于执行该节点的任务的恢复值列表。每当执行恢复时，它都从节点的开头开始。对于遇到的每个中断，LangGraph 都会检查任务的恢复列表中是否存在匹配的值。匹配是严格基于索引的，因此节点内中断调用的顺序非常重要。

  • ✅ 保持不同节点执行之间的 interrupt 调用一致
def node_a(state: State):
    # ✅ Good: interrupt calls happen in the same order every time
    name = interrupt("What's your name?")
    age = interrupt("What's your age?")
    city = interrupt("What's your city?")

    return {
        "name": name,
        "age": age,
        "city": city
    }
  • 🔴 不要在节点内有条件地跳过 interrupt 调用
  • 🔴 不要使用在不同执行之间非确定性的逻辑来循环 interrupt 调用

不要在 interrupt 调用中返回复杂值

根据使用的检查点记录器，复杂值可能无法序列化（例如，您无法序列化函数）。为了使您的图适应任何部署，最佳实践是仅使用可以被合理序列化的值。

  • ✅ 向 interrupt 传递简单的、可 JSON 序列化的类型
  • ✅ 传递具有简单值的字典/对象
  • 不要向 interrupt 传递函数、类实例或其他复杂对象

在 interrupt 之前调用的副作用必须是幂等的

例如，您可能在节点内有一个更新记录的 API 调用。如果在发出该调用之后调用了 interrupt，则在恢复节点时该调用将再次运行，可能会覆盖初始更新或创建重复记录。

  • ✅ 在 interrupt 之前使用幂等操作
  • ✅ 将副作用放在 interrupt 调用之后
  • ✅ 尽可能将副作用分离到单独的节点中

对作为函数调用的子图使用中断

在节点内调用子图时，父图将从调用子图且触发 interrupt 的节点开头恢复执行。类似地，子图也将从调用 interrupt 的节点开头恢复。

def node_in_parent_graph(state: State):
    some_code()  # <-- This will re-execute when resumed

使用中断进行调试

要调试和测试图，您可以将静态中断用作断点，以便一次一个节点地逐步执行图。静态中断在节点执行之前或之后的定义点触发。您可以通过在编译图时指定 interrupt_before 和 interrupt_after 来设置这些中断。

graph = builder.compile(
    interrupt_before=["node_a"],
    interrupt_after=["node_b", "node_c"],
    checkpointer=checkpointer,
)

# Pass a thread ID to the graph
config = {
    "configurable": {
        "thread_id": "some_thread"
    }
}

# Run the graph until the breakpoint
graph.invoke(inputs, config=config)

# Resume the graph
graph.invoke(None, config=config)
config = {
    "configurable": {
        "thread_id": "some_thread"
    }
}

# Run the graph until the breakpoint
graph.invoke(
    inputs,
    interrupt_before=["node_a"],
    interrupt_after=["node_b", "node_c"],
    config=config,
)

# Resume the graph
graph.invoke(None, config=config)
  1. 调用 graph.invoke 时带有 interrupt_before 和 interrupt_after 参数。这是一种运行时配置，可以为每次调用进行更改。
  2. interrupt_before 指定在执行节点之前应暂停执行的节点。
  3. interrupt_after 指定在节点执行后应暂停执行的节点。
  4. 图将运行直到遇到第一个断点。
  5. 通过为输入传入 None 来恢复图。这将运行图，直到命中下一个断点。
